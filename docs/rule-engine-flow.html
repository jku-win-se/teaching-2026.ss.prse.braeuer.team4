Two Threads, One Rule

JavaFX is single-threaded — UI components can only be touched from the JavaFX Application Thread. The rule engine needs a background timer but must update the UI. The solution: split the work across two threads.

🕐 Scheduler Thread — "jdbc-rule-dispatcher"
Created once in startRecurringExecution()
Uses a single-thread ScheduledExecutorService
daemon = true (JVM won't wait for it on shutdown)
Fires every 30 seconds
scheduleAtFixedRate(..., 0, 30, SECONDS)
Does NOT touch the UI or devices.
Only posts a task:
Platform.runLater(this::processDueRules)
Why daemon?
When the app closes, the JVM shuts down immediately without waiting for this thread to finish its current tick.
Platform
.runLater
🖥️ JavaFX Application Thread
The only thread allowed to touch UI.
All JavaFX controls (labels, lists, badges) must be updated here.
Picks up the task from the event queue
Platform.runLater() adds the task; JavaFX processes it between UI events
Runs processDueRules()
— loops over all rules
— calls RuleEvaluator
— calls applyAction()
— calls NotificationService (updates UI badge)
— calls LogService
Also handles user clicks
Button presses, form submissions — all on the same thread. This is why long-running work must NOT run here.
Why this split? If the scheduler thread tried to update the notification badge directly, JavaFX would throw an exception: "Not on FX application thread". The scheduler thread only knows how to post a task — it's like putting a note in a mailbox. The JavaFX thread reads the mailbox when it has a moment and does the real work.

The Code

public void startRecurringExecution() { ruleScheduler = Executors.newSingleThreadScheduledExecutor(threadFactory); ruleScheduler.scheduleAtFixedRate( () -> Platform.runLater(this::processDueRules), 0, // initial delay (seconds) 30, // repeat every 30 seconds TimeUnit.SECONDS); }
outer lambda → runs on scheduler thread
Platform.runLater() → posts to JavaFX queue
processDueRules() → runs on JavaFX thread

The Code

private void processDueRules() { RoomService roomSvc = ServiceRegistry.getRoomService(); RuleEvaluator evaluator = new RuleEvaluator(); // created ONCE (PMD: AvoidInstantiatingObjectsInLoops) for (Rule rule : rules) { if (!rule.isEnabled()) { activeRuleIds.remove(rule.getId()); // reset state for disabled rules continue; } Device src = roomSvc.getDeviceByName(rule.getSourceDevice()); boolean conditionMet = evaluator.evaluate(rule, src); if (conditionMet && !activeRuleIds.contains(rule.getId())) { activeRuleIds.add(rule.getId()); // mark: condition is NOW true executeEnabledRule(rule, false); // FIRE — false→true transition } else if (!conditionMet) { activeRuleIds.remove(rule.getId()); // condition gone → reset so it can fire again } // conditionMet AND already active → skip (no spam) } }

RuleEvaluator — Three Trigger Types

Stateless class with no service dependencies. Takes a rule and a device, returns true/false. Testable in isolation — no JavaFX, no database needed.

Condition format: "06:00 AM" or "Weekdays 06:00 AM". Returns true for the entire minute the clock matches.

private boolean evaluateTime(String condition) { LocalTime target = parseTime(condition); // supports 12h and 24h formats if (target == null) return false; // unparseable → safe default LocalTime now = LocalTime.now(); return now.getHour() == target.getHour() && now.getMinute() == target.getMinute(); // true for the full minute → edge detection ensures it fires exactly once }

private boolean evaluateDeviceState(String condition, Device src) { if (condition == null || src == null) return false; // null-safe String n = condition.trim().toLowerCase(); if (n.equals("state = active") || n.equals("state = on")) return src.getState(); // true if device is on if (n.equals("state = inactive") || n.equals("state = off")) return !src.getState(); // true if device is off return false; }

private boolean evaluateSensorThreshold(String condition, Device src) { String[] parts = condition.trim().split("\\s+"); // ["Value", ">", "25"] if (parts.length != 3 || !parts[0].equalsIgnoreCase("Value")) return false; double threshold = Double.parseDouble(parts[2]); double sensorValue = src.getCurrentValue(); return switch (parts[1]) { case ">" -> sensorValue > threshold; case ">=" -> sensorValue >= threshold; case "<" -> sensorValue < threshold; case "<=" -> sensorValue <= threshold; case "=", "==" -> Double.compare(sensorValue, threshold) == 0; default -> false; }; }

Scheduled vs Manual Execution

🕐 Scheduled (automatic every 30s)

// from processDueRules() executeEnabledRule(rule, false); // inside executeEnabledRule: conditionMet = false || evaluator.evaluate(rule, src); // condition MUST be true to proceed

Edge detection in processDueRules() already ensures this is only called on false→true transition.

👆 Manual (user clicks "Run" in UI)

// from executeRule(id, isManual=true) executeEnabledRule(rule, true); // inside executeEnabledRule: conditionMet = true || evaluator.evaluate(rule, src); // short-circuits: fires regardless of condition

Bypasses activeRuleIds too — manual run is always allowed even if the rule already fired this cycle.
