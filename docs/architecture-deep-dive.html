Architecture in 6 Layers

The app follows a strict top-to-bottom dependency rule: each layer only knows about the layer directly below it. The View never touches the database. The Controller never knows whether it's talking to a mock or a real database — it only talks to an interface.

How a Controller loads its service

A Controller never calls new JdbcRuleService(). It always asks the ServiceRegistry. This is the key to testability.

// Field initializer in RulesController.java private final RuleService ruleService = ServiceRegistry.getRuleService(); // ↑ Interface ↑ Returns JdbcRuleService in production // Returns MockRuleService in tests // Usage — Controller doesn't know or care which implementation it got private void handleAddRule() { Rule rule = ruleService.addRule(name, triggerType, source, cond, action, target); if (rule != null) refreshTable(); }

The Holder Pattern — Thread-Safe Lazy Init (PMD Finding #1)

❌ Before — Double-Checked Locking (broken)
private static volatile RuleService svc; public static RuleService getRuleService() { if (svc == null) { synchronized (ServiceRegistry.class) { if (svc == null) { svc = JdbcRuleService.getInstance(); } } } return svc; } // ⚠ volatile is needed but the pattern // is still error-prone and hard to reason about
✅ After — Initialization-on-Demand Holder
// No volatile, no synchronized needed here public static RuleService getRuleService() { return testOverride != null ? testOverride : RuleServiceHolder.INSTANCE; } private static final class RuleServiceHolder { private static final RuleService INSTANCE = JdbcRuleService.getInstance(); } // JVM guarantees the class is loaded exactly // once, thread-safely, on first access

Mock vs JDBC — Two Implementations, One Interface

Every service has two implementations. This is the core design decision that makes the codebase testable without a running database.

Mock Implementations
📦
Storage: Plain Java lists and maps in memory. No files, no network.
⚡
Speed: Microseconds. Tests run instantly — no network roundtrip.
🔄
Reset: resetForTesting() clears all state between test cases — no data bleeds between tests.
🌱
Seed data: initializeMockRules() pre-populates demo data so the UI looks populated from the start.
🔬
Used in: All unit tests, controller tests, coverage tests.
⚠️
Limitation: Data is lost when the app restarts. Not suitable for production.
JDBC Implementations
🗄️
Storage: Aiven cloud PostgreSQL. All data survives restarts and is shared across sessions.
🔒
Schema init: Each service runs its own init-xxx.sql on first connection to create tables if they don't exist.
🔁
In-memory mirror: An ObservableList mirrors the DB table so JavaFX UI bindings update live without re-querying.
🌱
Seed data: seedDemoRules() etc. inserts demo data only if the table is empty on first run.
🏭
Used in: Production app. Also used in integration tests with H2 in-memory DB.
🔑
Config: DB URL from .env / env vars / smarthome-db.properties.

App Startup Sequence

What happens from double-clicking the JAR to seeing the login screen. The purple steps are new fixes we added today.

JVM
Java launches the application, calls SmartHomeApp.main(args)
Application.launch(SmartHomeApp.class, args)
JavaFX Runtime
JavaFX initializes and calls SmartHomeApp.start(primaryStage) on the JavaFX Application Thread
Stage Setup
Window title, size, and minimum dimensions are set. No services touched yet.
primaryStage.setTitle("Smart Home Orchestrator"); setWidth(1000); setHeight(700)
Load Login + Register scenes
Two lightweight FXML files are parsed. Their controllers are created — these only hold UI references, no service calls yet.
loadLoginScene(); loadRegisterScene();
Show Window
The login screen appears immediately. User can start reading the form.
primaryStage.setScene(loginScene); primaryStage.show();
User sees the app instantly
Platform.runLater() — deferred
The heavy work is posted to the JavaFX event queue instead of running before show(). This is the startup fix — previously these lines came before show() and blocked the window from appearing.
Platform.runLater(() -> { loadMainScene(); startServices(); })
Fixed today — was the cause of the freeze
loadMainScene()
main-shell.fxml is loaded. MainController is created. Its field initializers trigger ServiceRegistry.getNotificationService() → lazy-initializes JdbcNotificationService → opens first connection to Aiven PostgreSQL. This takes 1–3 seconds.
ServiceRegistry.getNotificationService() → JdbcNotificationServiceHolder.INSTANCE
Network round-trip to Aiven happens here
Start Background Services
Schedule and Rule recurring execution starts. A daemon thread wakes every 30 seconds and posts processDueRules() via Platform.runLater().
getScheduleService().startRecurringExecution(); getRuleService().startRecurringExecution();
App ready
User can now log in. On successful login, showMainScene() switches to the main shell. If mainScene is still null (init not finished), it retries via Platform.runLater(this::showMainScene).
