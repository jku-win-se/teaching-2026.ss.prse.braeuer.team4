Key Design Decisions

Initialization-on-Demand Holder

Thread-safe lazy init for all JDBC services — no explicit locks, no double-checked locking (→ PMD Finding #1). Each service lives in its own static inner Holder class.

Test Injection via ServiceRegistry

ServiceRegistry.setXxxForTesting(mock) replaces any service at runtime. Tests never touch production JDBC — they inject mocks before each test method.

Startup Fix (today)

JDBC init was blocking the JavaFX thread before show(). Fixed with Platform.runLater() so the login window appears immediately.

Graceful MQTT Degradation

All FR-01..FR-17 features work without a broker. Integration devices show OFFLINE. No retry loop blocks the UI thread.

PMD Findings

1
DoubleCheckedLocking
Bug ▼

ServiceRegistry used the double-checked locking pattern — a well-known broken concurrency anti-pattern in Java. Even with volatile, it's error-prone and hard to reason about.

❌ Before
private static volatile ScheduleService service;

public static ScheduleService get() {
  if (service == null) {
    synchronized (ServiceRegistry.class) {
      if (service == null) {
        service = new JdbcScheduleService();
      }
    }
  }
  return service;
}
✅ After
public static ScheduleService get() {
  return Holder.INSTANCE;
}

private static final class Holder {
  private static final ScheduleService
    INSTANCE = JdbcScheduleService
      .getInstance();
}
Takeaway: Initialization-on-demand Holder — thread-safe lazy init with zero synchronization overhead, guaranteed by the JVM class loader.
2
SystemPrintln
Bug ▼

A debug leftover System.out.println was dumping the entire CSV export to stdout in production. Users never see it — but it clutters logs and can leak data.

❌ Before
private void handleExportCSV() {
  String csv = logService.exportToCSV();
  // "In a real app, save to file..."
  System.out.println(
    "Export CSV:\n" + csv  // BUG
  );
}
✅ After
private void handleExportCSV() {
  logService.exportToCSV();
  // save to file / clipboard
}
Takeaway: PMD acts as a safety net — catches debug code before it ships.
3
AvoidCatchingNPE
Bug ▼

NullPointerException was used for control flow when loading FXML resources. This catches any NPE in the try block — not just the intended one — and is slow and misleading.

❌ Before
try {
  FXMLLoader l = new FXMLLoader(
    Objects.requireNonNull(
      getClass().getResource(path))
  );
  Node v = l.load();
} catch (NullPointerException e) {
  showError("Not found", e); // BAD
}
✅ After
URL res = getClass().getResource(path);
if (res == null) {
  showError("Not found");
  return;
}
FXMLLoader l = new FXMLLoader(res);
Node v = l.load();
Takeaway: Explicit null checks are always better than catching NPE — they're faster, clearer, and don't accidentally swallow unrelated bugs.
4
NullAssignment → domain logic bug
Bug ▼

Session expiry called logout() which set currentUserRole = null. The rest of the code expected "Guest" — not null. PMD's NullAssignment flag revealed a real domain logic error.

❌ Before
if (System.currentTimeMillis()
    >= sessionExpiresAt) {
  logout();
  // sets currentUserRole = null!
}
✅ After
if (System.currentTimeMillis()
    >= sessionExpiresAt) {
  currentUserEmail = null;
  currentUsername = null;
  currentUserRole = "Guest"; // ✓
  currentUserStatus = null;
  sessionExpiresAt = 0;
}
Takeaway: PMD's NullAssignment warning triggered investigation — revealing a domain logic bug where an expired session produced a null role instead of "Guest".
5
OnlyOneReturn
Design ▼

ActivityLogController.matches() had 4 early return statements. Restructuring with named boolean variables makes the logic immediately readable.

❌ Before
if (filterByDevice &&
    !device.equals(entry.getDevice()))
  return false;
if (fromIso != null &&
    date.compareTo(fromIso) < 0)
  return false;
// ... more returns ...
return true;
✅ After
final boolean deviceOk = !filterByDevice || device.equals(entry.getDevice()); final boolean fromOk = fromIso == null || date.compareTo(fromIso) >= 0; final boolean toOk = ...; return deviceOk && fromOk && toOk;
Takeaway: Named boolean variables turn implicit logic into a self-documenting expression.
6
ShortVariable
Style ▼

Single-letter variable names like d, r, t, s were used throughout. A reader has to trace back to the declaration to understand what each one means.

❌ Before
String id = UUID.randomUUID()...;
String t = type.trim().toLowerCase();
String s = sql.trim();
Room r = input.room();
Device d = roomSvc.addDevice(...);
✅ After
String deviceId = UUID.randomUUID()...; String normalizedType = type.trim()...; String sqlStatement = sql.trim(); Room selectedRoom = input.room(); Device createdDevice = svc.addDevice(...);
Takeaway: Short names save typing, but cost reading time. PMD enforces intention-revealing names.
9
InsufficientStringBufferDeclaration
Performance ▼

CSV export created a StringBuilder with default capacity 16. With potentially hundreds of log entries (~96 chars each), the internal array was reallocated many times.

❌ Before
StringBuilder csv = new StringBuilder(); // capacity = 16 → many reallocations csv.append("Timestamp,Device,..."); for (LogEntry e : logs) { ... }
✅ After
int cap = logs.size() * 96 + 64; StringBuilder csv = new StringBuilder(cap); // one allocation, no copies csv.append("Timestamp,Device,...");
Takeaway: Pre-sizing a StringBuilder avoids O(n) reallocations. PMD catches this automatically.
11
UnitTestContainsTooManyAsserts
Design ▼

A single test method was checking 7+ things at once. When it failed, you couldn't tell which assertion broke.

❌ Before (1 test, 7 asserts)
@Test void addUpdateToggleDelete() { Schedule s = svc.addSchedule(...); assertNotNull(s); assertEquals(1, svc.size()); assertTrue(svc.update(...)); assertEquals("New", s.getName()); // ... 3 more ... }
✅ After (focused tests)
@Test void addReturnsNonNull() {...} @Test void addIncreasesSize() {...} @Test void updateChangesName() {...} @Test void toggleEnablesInactive() {...} @Test void deleteReducesSize() {...}
Takeaway: One assert per test = precise failure localization. When the test fails, the name tells you exactly what broke.
12
MethodNamingConventions — suppress with reason
Judgment ▼

PMD expected camelCase for all methods, but test methods using snake_case are a widely accepted convention for readability. This is a case where blind compliance would reduce code quality.

PMD wanted: camelCase
@Test void freshDatabaseHasNoNotifications() { // harder to read at a glance }
✅ Kept + suppressed
@SuppressWarnings("PMD.MethodNamingConventions") class TestJdbcRuleService { ... } @Test void freshDatabase_hasNoNotifications() { // instantly readable }
Takeaway: Not every PMD finding needs a code change. When a rule conflicts with an established convention, suppress with justification — don't blindly comply.
