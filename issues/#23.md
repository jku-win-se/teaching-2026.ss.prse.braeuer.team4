FR-14 — Energy consumption dashboard

Related FR: FR-14 · User Stories: US-25, US-26 · Milestone: 2
Context

We have an EnergyController and a MockEnergyService skeleton, but no aggregated view. The dashboard must turn raw per-device estimates into a readable summary at three levels (device → room → household) over two time scales (daily, weekly), so a user can spot which device or room is driving consumption.
User Story

    As a user, I want to view an energy dashboard that shows estimated power consumption per device, per room, and for the whole household, aggregated by day and by week, so that I can identify high-consumption devices and adjust my usage.

Acceptance Criteria

    Estimation model
        Each device type has a documented nominal power (W) used for estimation; values live in one constants file and are referenced from a single place.
        Consumption (Wh) per device per period = nominal power × on-time during that period (derived from the activity log).
    Dashboard view
        Given the user opens Energy, then three panels are visible:
            By device — list of all devices with their daily and weekly Wh.
            By room — list of all rooms with their daily and weekly Wh (sum of contained devices).
            Whole household — single card with daily and weekly Wh totals.
        All values are formatted in Wh or kWh with at most 2 decimals.
    Time scale
        "Daily" = current calendar day (00:00 → now).
        "Weekly" = current ISO week (Mon 00:00 → now).
        The user can switch between today, yesterday, this week, last week.
    Empty / zero data
        Given a device has no activity in the period, then it appears with 0 Wh (not hidden, not error).
    Reactive update
        When a device state change is logged, the relevant aggregate refreshes within ≤ 2 s without a manual reload.

Out of Scope

    Real metered consumption from physical devices (FR-18, FR-18 The system shall support an optional integration layer for a physical IoT protocol (e.g., MQTT) to connect real devices. #27).
    Cost estimation in EUR and tariff handling.
    Long-term history beyond "last week".

Technical Notes

    Add EnergyService.getDailyByDevice(LocalDate), getWeeklyByDevice(int isoWeek) and equivalents for room/household. Keep the controller thin.
    Source of truth for on-time = LogEntry stream; do not duplicate state.
    Aggregation is pure: write it as testable functions independent of JavaFX.

Definition of Done

    mvn verify green.
    Unit tests cover: known on-time → expected Wh, room sum equals device sum, household sum equals room sum, zero-activity case.
    Manual smoke: toggle a device on for a known interval and verify the displayed Wh matches the formula (within rounding).
    Dashboard refreshes after a state change without manual reload.
    Code reviewed; PR merged.
