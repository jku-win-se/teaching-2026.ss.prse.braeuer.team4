FR-15 — Detect and warn about scheduling conflicts

Related FR: FR-15 · User Story: US-27 · Milestone: 2
Context

With several rules and schedules per device, two automations can quietly contradict each other (e.g. rule A turns the living-room light on at 20:00). Today the system silently runs whichever happens last. We need to detect contradictions at save time and warn the user, so the household doesn't end up with flickering or "stuck" devices.
User Story

    As a user, I want the system to detect conflicts where two rules would put the same device into contradictory states at the same time, and warn me before saving, so that I can correct invalid automations before they cause problems.

Acceptance Criteria

    Conflict definition
        A conflict exists when two enabled rules target the same device with incompatible target values that overlap in time.
        "Incompatible" is documented per device type:
            SwitchDevice: ON vs OFF.
            DimmerDevice: brightness values that differ by more than 0.
            ThermostatDevice: target temperatures that differ by more than 0.5 °C.
            CoverDevice: OPEN vs CLOSED.
    At save time
        Given the user creates or edits a rule, when they hit Save, then the system checks every other enabled rule for the same device and the same time window.
        When a conflict is found, then a modal warning lists each conflicting rule by name, target value, and trigger time, and offers one choices: Cancel.
    No false positives
        Two rules that target the same device with the same value at the same time are not flagged.
        Two rules with disjoint time windows are not flagged.

Out of Scope

    Auto-resolving conflicts (system picks a winner).
    Conflicts across scenes (FR-17, FR-17 The system shall provide a "Scenes" function: a named group of device states that can be activated with a single action (e.g., "Movie Night" dims the lights and closes the blinds). #26) — covered separately if needed.

Technical Notes

    New ConflictDetectionService with one public method detect(Rule|Schedule candidate): List<SchedulingConflict>.
    Time-window comparison helper takes two recurrence patterns and returns whether they overlap; keep it pure for unit testing.
    Reuse the SchedulingConflict class already in docs/Grobkonzept Domain Model.mmd.

Definition of Done

    mvn verify green.
    Unit tests cover: each device-type conflict matrix, disjoint windows pass, identical-value windows pass, overlapping incompatible windows fail.
    UI smoke: deliberately create a conflicting rule and observe the modal + the warning icon; resolve it and confirm the icon disappears.
