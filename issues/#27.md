FR-18 — Optional MQTT integration for real devices

Related FR: FR-18 · User Story: US-31 · Milestone: —
Context

The system currently only manages virtual devices. FR-18 introduces an optional integration layer so a real device can be paired via MQTT and appear next to virtual ones. The skeleton (model/IntegrationDevice.java, controller/IoTSettingsController.java) is in place but no broker connection or message routing exists. Because this involves an external dependency, the integration must be opt-in: the system must run normally with the broker turned off.
User Story

    As a user, I want to connect supported physical smart devices through an MQTT integration layer, so that I can control real devices alongside the simulated ones from the same UI.

Acceptance Criteria

    Configuration
        Given the user opens Settings → IoT, then they can enter broker host, port, optional username/password, and an Enable integration toggle. Default is off.
        When the toggle is on and Test connection is clicked, then the UI shows Connected or a clear error message within ≤ 5 s.
    Pair a real device
        Given integration is enabled and connected, when the user adds an IntegrationDevice (name, type, MQTT topic), then the device subscribes to its state topic and appears in the Devices view with a small "real" badge.
    Bidirectional control
        User-initiated state changes from the UI are published to the device's command topic.
        State updates received on the state topic update the device's displayed state within ≤ 2 s.
    Graceful degradation
        Given the broker is unreachable, when the app starts, then virtual devices and all FR-01..FR-17 features still work; integration devices show status OFFLINE.
        No retry loop blocks the UI thread.
    Permissions
        Only Owner can configure the broker or pair / unpair real devices (consistent with FR-13, FR-13 The system shall support two user roles: Owner (full access) and Member (control only, no device/rule management). #22).
    Logging
        Every published or received message produces an activity-log entry (actor = MQTT, includes topic and payload summary).

Out of Scope

    Provisioning real devices via Bluetooth / Zigbee / Matter.
    TLS client certificates (username + password is enough for M3).
    Auto-discovery of devices on the broker (Home Assistant style).

Technical Notes

    Use the Eclipse Paho Java client (already common in JKU course examples) — single dependency in pom.xml.
    Encapsulate broker access behind one MqttIntegrationService so the rest of the app depends on a small interface, not the Paho API. This also makes a MockMqttIntegrationService trivial for tests.
    Topic convention: smarthome/<deviceId>/state and smarthome/<deviceId>/cmd. Document this in docs/.

Definition of Done

    mvn verify green; runs with broker disabled.
    Unit tests on MqttIntegrationService use a MockMqttClient (no real network).
    Manual smoke against a local Mosquitto: pair one device, change state from UI → see broker message; publish a state from mosquitto_pub → see UI update.
    Connection failure scenario: broker stopped → app still usable, device shown OFFLINE.
    Code reviewed; PR merged.
    docs/user-stories.md US-31 marked as implemented.
    docs/system-architecture.md updated with the topic convention and the integration toggle.
